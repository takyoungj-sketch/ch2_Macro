# 토지 데이터 정제 기준

이 문서는 `참고/7.토지 통합 정제.ipynb`의 핵심 정제 과정을 서비스 파이프라인에 반영하기 위한 기준이다. 웹서비스의 `pipeline/clean.py`는 아래 기준으로 `land_transactions`를 만들고, `pipeline/build_stats.py`는 이 정제 결과를 기준으로 `land_basic_stats`를 생성한다.

## 입력 통합

- 국토부 토지 매매 Excel 원본은 `skiprows=13`, `header=None`으로 읽고 서비스 표준 컬럼으로 맞춘다.
- 통합 파일은 원본 행을 이어 붙인 헤더 없는 Excel로 취급한다.
- 서비스의 `pipeline/collect.py --mode excel --format raw|merged|auto`는 이 입력 방식을 담당한다.

## 핵심 컬럼

정제 후 분석에 사용하는 핵심 컬럼은 다음이다.

- 지역: `sigungu_name`, `beopjungri_code`, `sido_code`, `sigungu_code`
- 토지 특성: `land_category`, `zone_type`, `road_condition`
- 계약: `contract_year`, `contract_month`, `contract_date`
- 면적·가격: `area_sqm`, `area_category`, `total_price_10k`, `unit_price_per_sqm`
- 거래 상태: `is_partial_ownership`, `is_cancelled`, `is_valid`

## 필터와 파생값

- 거래금액은 콤마를 제거한 뒤 숫자로 변환한다.
- 계약면적은 숫자로 변환하고, 숫자가 아니거나 `-`, `*`, 공백이면 결측으로 처리한다.
- 해제사유발생일, 해제유형, 해제여부에 실제 값이 있으면 해제거래로 본다. `-`, `*`, 공백 등 부재 표기는 미해제로 본다.
- `is_valid`는 면적과 금액이 정상이고 해제거래가 아닌 행만 `TRUE`로 둔다.
- 동일 물건에 정상 신고와 해제 신고가 함께 존재할 수 있으므로, 정제 테이블은 신고 행 단위로 보존한다. 해제 신고는 통계에서 제외하되, 같은 물건의 정상 신고를 함께 제거하지 않는다.
- 신고 행 고유키는 원본 `순번`을 우선 사용하고, 순번이 없는 수집 경로에서는 `raw_id`를 fallback으로 사용한다. 정상 신고와 해제 신고가 UPSERT 과정에서 서로 덮어쓰지 않게 하기 위함이다.
- 면적구분은 계약면적 기준으로 만든다.
  - `광소`: 30㎡ 미만
  - `정상`: 30㎡ 이상 3000㎡ 미만
  - `광대`: 3000㎡ 이상
- 단가는 엑셀 정제 파일과 동일하게 `거래금액(만원) / 계약면적(㎡)`로 계산하고 소수점 1자리로 반올림한다. 저장 컬럼은 기존 스키마 호환을 위해 `unit_price_per_sqm`을 쓰지만 단위는 `만원/㎡`이다.

## 재생성 절차

이미 적재된 DB는 과거 정제 기준으로 만들어졌을 수 있다. 위 기준을 반영하려면 원자료 기준으로 다시 정제하고 사전집계를 다시 만들어야 한다.

1. `land_transactions_raw`에 원본이 존재하는지 확인한다.
2. 기존 `land_transactions`와 `land_basic_stats`는 백업 후 재생성한다.
3. `pipeline/clean.py --reprocess-all`로 이미 처리된 raw도 다시 정제해 `land_transactions`를 갱신한다.
4. `pipeline/build_stats.py`로 `land_basic_stats`를 다시 만든다.
5. 같은 지역·기간의 엑셀 분석표와 거래건수, 단가, 용도지역×지목 조합을 대조한다.

정제 기준 변경 후에는 기존 거래 고유키와 새 신고 행 고유키가 공존할 수 있으므로, 운영 DB에서는 백업 후 `land_transactions`, `land_basic_stats`를 비우고 재생성하는 절차가 가장 안전하다.

## 2026-05-10 검증 이력

호암동 `2022~2025` 자료로 참고 엑셀 정제 결과와 서비스 DB를 대조했다.

- 참고 엑셀 정제 CSV: `110건` (`2022`년 단일 연도 검증 파일)
- DB export 전체: `111건`
- DB export 중 `is_valid = TRUE`, `is_cancelled = FALSE`: `106건`
- 차이 원인: 같은 물건에 정상 신고와 해제 신고가 함께 존재하는 경우, 기존 거래 해시가 두 신고를 같은 거래로 보아 해제 신고가 정상 신고를 덮어썼다.

전체 호암동 `2022~2025` raw 기준 검증:

- raw 원자료: `433건`
- 해제 제외 유효 신고: `406건`
- 이전 물건 단위 해시 기준 고유 거래: `366건`
- 기존 집계 반영: `363건`
- 누락 원인:
  - `40건`: 동일 물건/동일 조건 신고가 하나의 거래 해시로 합쳐짐
  - `3건`: 동일 물건의 해제 신고가 정상 신고를 덮어써 집계에서 제외됨

최종 확정 정책:

- 정제 테이블은 **신고 행 단위**를 보존한다.
- 해제 신고는 `is_cancelled = TRUE`, `is_valid = FALSE`로 두고 통계에서 제외한다.
- 같은 물건에 정상 신고와 해제 신고가 함께 있더라도 정상 신고는 보존한다.
- `transaction_hash`는 원본 `순번`을 우선 포함하고, 순번이 없으면 `raw_id`를 포함한다.
