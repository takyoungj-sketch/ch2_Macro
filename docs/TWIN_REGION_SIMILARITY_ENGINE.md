# Twin Region Similarity Engine — 설계안 (수정본)

토지 실거래·인구 등 **현재 레포 보유 데이터만** 사용해 행정구역(주로 시군구·읍면동)의 시장 구조를 벡터화하고, **인접 광역 행정기준** 안에서 후보를 제한해 유사 지역(**Twin Region**)을 찾아 주는 기능의 MVP 설계를 정리한다. 본 기능은 실시간 API가 아니라 **월간 배치 파이프라인**의 `build_stats` 계열 후반 단계에서 산출·적재한다.

---

## 1. 프로젝트 목표

- 토지 실거래 원장 및 사전 집계를 활용해 **행정구역별 시장 fingerprint**를 만든다.
- 인구·거래 패턴·용도지역×지목 구조를 바탕으로 **유사한 쌍·Top‑k 후보**를 계산한다.
- 결과를 **PostgreSQL**에 적재하여 이후 클라이언트(레이더·설명 카드 등)에서 소비한다.
- 초기 검증에서는 **설명 가능성과 QA**가 스코어 순위 자체보다 우선한다.

---

## 2. 핵심 설계 철학

본 시스템은 단순 가격 비교가 아니다.

- **토지 시장 구조의 유사성**(용도×지목 믹스 방향성)
- **거래 패턴의 방향성**(비중 패턴 위주 해석)
- **용도지역 × 지목 기반 시장 fingerprint**

을 비교 목표로 둔다. 개념적으로는 공간 속성 하나가 아니라 **지역 임베딩(region embedding)** 에 가까운 레이어마다, MVP에서는 해석 가능한 명시적 피처 공간으로 구현한다.

---

## 3. 현 단계 원칙

### 3.1 외부 데이터 추가 금지 (MVP 고정)

초기 MVP에서는 **레포 및 기존 DB·파이프라인 산출물로 이미 존재하는 데이터만** 사용한다.

**추가하지 않는 데이터 예시**(비목록이 아니라 정책 예시이다):

- 사업체수, 소득, 유동인구, 카드 소비 등
- 건축허가·자동차등록 등 행안부 계열 신규 파이프

**허용하는 데이터**(현 레포 현실에 맞게 조정 가능):

- 토지 실거래 원장 및 이로부터 계산되는 건수·면적·단가 계열 지표
- `region_codes` 및 인구 계열 테이블(기존 연말 인구 등)
- 용도지역(`zone_type`)·지목(`land_category`) 등 원장 속성

**목적**: 외연 없이 **현 데이터만으로 설명 가능한 한도** 안에서 유사도의 신뢰를 검증한다.

### 3.2 실시간이 아닌 월간 배치

- Twin 산출은 **`build_stats` / V2 통계 재생 성공 직후** 또는 동일 묶음의 명시 단계에서 수행한다.
- 동일 **`as_of_month`(또는 스냅샷 버전)** 기준으로 다른 사전 집계와 정합성을 맞춘다.

### 3.3 Twin은 Regional Profile을 「소비」한다 — Feature 재생성 금지 (D-017)

계층 순서는 **Market Stats → Regional Profile → Twin** 이다 ([`REGIONAL_PROFILE_ARCHITECTURE.md`](REGIONAL_PROFILE_ARCHITECTURE.md) Layer 5).

- Twin은 **상위 소비자(서비스 계층)** 이며, 용도지역×지목 비중·가격 수준 등 Feature를 **다시 만들지 않는다.**
- 입력은 `SELECT * FROM regional_profile WHERE profile_version=:v AND as_of_month=:as_of AND window_years=:w` 로 **고정 버전·스냅샷**을 읽는다.
- 본 문서 §5의 「구조 블록/가격 블록」은 **Profile feature로부터의 변환·결합 규약**으로 해석한다. 원천 집계는 Profile/Market Stats가 **SSOT**이며 Twin 빌더에 이중 구현하지 않는다.
- 따라서 Twin 결과는 `(profile_version, as_of_month, window_years)` 에 **종속**되며, `detail_scores`·아티팩트에 이 3개 키를 함께 기록한다(재현성).

## 4. 탐색 범위 (Search Space)

### 4.1 인접 행정기준만 허용

비교 후보 후보수를 제어하고 행정·지리적 무관 비교 지양을 위해:

- **경계가 맞닿는 시·도(또는 정책이 정하는 광역 인접 쌍)** 사이만 유사도 탐색을 허용한다.
- **직선거리·Haversine 기반 근처 탐색은 사용하지 않는다.**

실 구현에서는 반드시 **사전 구축된 인접 룩업 테이블**(예: `sido_adjacency` 또는 레벨별 adjacency 맵 + 유효기간 버전 필드)을 사용하고, 버전 변경 시 재생산 절차를 문서화한다.

**예**(정책 설명용):

- 충북 ↔ 경기: 인접 쌍 테이블에 있으면 후보 허용
- 충북 ↔ 제주: 인접가 아니면 후보 미생성(MVP에서는 탐색 대상 외부)

### 4.2 장기 방향(문서 고정만)

실무에서는 비인접이어도 구조적으로 쌍이 되는 경우가 있다. 장기에는 **선택 플래그로 인접 제한 완화**(같은 시도 안, 동일 규모권 등)를 두는 확장 가능성만 남겨 두고 MVP에서는 적용하지 않는다.

---

## 5. 유사도 계산 구조

### 5.1 1단계: 체급 필터링 (인구)

- **설정 가능한 허용 범위**로 인구 차이 큰 후보를 제외한다.

**예시 파라미터:**

```text
POP_TOLERANCE_REL = 0.2    # 현재 후보 인구와 ±20%
```

동일 레벨(시군구 vs 읍면동) 간 해석 차이가 있어, 레벨별로 파라미터를 분리하거나 상위 레벨에서만 적용할지 명시 검토값으로 둘 수 있다.

**참고(선택 과제)**: 상·하 규모 쌍 형제 검색이 필요하면, 후속으로 **순위/분위수 기준 차이 허용** 등을 같은 인구 축 안에서 검토만 문서 상에 남긴다(MVP에서는 필수 아님).

### 5.2 2단계: 시장 구조 유사도 (메인 블록)

#### A. 거래건수 비중 벡터 (Primary)

- 차원 정의: **용도지역 × 지목** 조합별 **거래 건수 비중**(해당 레벨·기간 안에서 전체 거래건수 대비 분율).
- 예 라벨(개념): `자연녹지×답`, `계획관리×전`, `2종일반주거×대`, … 실제 문자열 규격은 빌더에서 일원화.
- 이 벡터를 **설명 우선 순위 최상단** 피처로 취급한다.
- 거래 건수가 존재하지 않는 조합은 **시장 부재**로 해석하며 해당 차원값은 **`0`(fill)** 으로 둔다. **차원 차원별 NULL 유지 금지**로 통일한다(`fillna(0)` 정책).

**유사도(구조 블록)**:

- **Cosine Similarity만 권장**한다. Euclidean 유사 거리는 같은 비중 블록에 대해 채택하지 않는 것을 원칙으로 한다(설계 결정 명시 목적).

#### B. 가격 수준 블록 (Secondary, 명시적으로 분리 결합 권장)

평균 단가(mean)만 사용하는 것은 **이상치 민감도** 때문에 MVP에서 채택하지 않는 것을 원칙으로 한다.

허용 예시 후보 중 하나 또는 조합을 **설명 가능한 이름**과 함께 보존한다:

- 셀 단위 중앙값(median 단가 등)
- 절단 평균(trimmed mean)
- `np.log1p(median_price)` 등 로그안정화 후 스칼라/소구간 피처

**중요(수정안 반영)**: **비중 벡터와 로그 단가 차원을 한 벡터에 이어 붙인 뒤 단일 cosine**으로 처리하면 차원별 스케일·의미가 섞여 **암묵적 가중**이 발생한다.

권장 결합 패턴:

1. **구조 블록**에 대해서만 cosine `sim_struct`(주된 지표).
2. 가격 수준 블록에 대해서는 **레벨·기간별로 하나 이상 스칼라**를 만들고 `sim_price`를 정의(예: z‑score 된 값들 간 상관 또는 별도 1‑차원 유사 함수).
3. 최종 순위 산술결합 예: `combined = w_struct * sim_struct + w_price * sim_price`(가중치는 설정 파일·레벨별).

이유는 Euclidean 금지가 아니라 **해석 책임**을 분리하기 위함이다.

#### C. 결측·희소·최소 표본 `MIN_TX_COUNT`

- 블록 A의 희소성은 cosine과 sparse CSR 형태와 잘 맞는다.
- **거래 규모가 너무 작은 지역**은 노이즈 가능성이 크므로 **최소 거래건수 threshold** 적용한다.

예시 설정:

```text
MIN_TX_COUNT_REGION = 30   # 정책 변수. 레벨별로 다른 값 허용
```

**중요**: 시군구와 읍면동을 동일 숫자로 묶으면 농산·변두리에서 전부 미산출된다. 따라서 **`region_level`(또는 비교 레이어)·집계 윈도**(예: 최근 12개월)·`as_of_month` 조합별로 다른 threshold 허용**을 설계 노트로 명시한다.

조건 미달 지역 처리(MVP 우선값):

- **유사도 계산에서 해당 지역을 제외**한다(집약적이고 QA하기 쉬움).  
  보조 패널티만 부여하는 방식은 이후 과제로 둘 수 있다.

#### D. 비중만으로는 차이 정보가 거의 없는 쌍 보조 보고

양쪽 조합 칸 모두 실질적인 비중이 0인 칸 위주여서 cosine이 과도하게 느긋하게 나오는 경우를 줄이기 위해, 선택적으로 다음을 `detail_scores`에 넣는다:

- **공통 지원 집합 유사성**(예: non‑zero 차원에 대한 **Jaccard** 또는 카운트 겹침비)

이 필드는 **순위를 바꾸기보다 QA·설명용** 우선 목표로 두어도 된다.

---

## 6. 정규화

스케일 차이 최소화:

- **블록 A(비중)**: 차원별 합이 1 또는 상수 규격이 되도록 정규화 정책을 한 줄로 고정. 보통 블록 내부에서 이미 분율이므로 cosine 직 입력 가능.
- **블록 B(가격 차원 스칼라·소벡터)**: `StandardScaler` 등 학습 가능한 scaler를 **매 스냅샷 학습**(해당 레벨 후보 세트 또는 전후보 피처로 fit) 규약을 명시한다. 결과 파라미터는 아티팩트로 저장 가능.

---

## 7. 성능·메모리

- 전국 읍면동 pairwise full scan 금지.
- **인접 제한 + 인구 필터 + sparse**(권장: `scipy.sparse.csr_matrix`)로 차원 처리.
- **중간 결과 재사용**:

**월간 parquet 예시**(이름 변경 가능):

| 필드 예시 |
|-----------|
| `region_code`(정책이 정하는 키) |
| `region_level`(sigungu / eupmyeondong 등 고정 문자열 토큰) |
| `as_of_month`(또는 동일 버전 키) |
| 구조 블록 sparse 또는 직렬화 벡터 |
| 선택 가격 블록 스칼라·소벡터 |
| 원시 `tx_count`, `population` |
| 선택 메타(스케일러 버전 등) |

Similarity 재계산 시 **pivot를 매번 새로 깊게 파지 않도록** parquet을 근거로 한다.

전체 결과를 모든 쌍 저장하지 않고 **후보별 인접·인구 거른 뒤 k‑NN**(또는 상한 인접 크기 후 코사인)으로 상한 시간을 두는 접근 가능.

## 8. DB 설계 방향

### 8.1 역할 분리(권장)

이름 예시이다. 마이그레이션 작성 시 DDL에서 확정한다.

1. **`twin_region_features`**(또는 동등 이름)  
   - 스냅샷마다 레벨·지역·feature 요약 하나의 행(또는 blob/JSON 규격 명시된 컬럼).
2. **`twin_region_neighbors`**(또는 `twin_similarity_rankings`)  
   - 레벨·지역·`as_of_month`별 **상위 Top‑k**(문서에서는 **Top 5** 예시)·score·detail.

이렇게 나누면 “피처 스토어”와 “매칭 결과”가 코드 검토 및 재계산 디버깅에서 분리된다. 단일 테이블에 모두 두는 초기안도 가능하나 명명충돌 가능성 증가.

### 8.2 `detail_scores`(JSONB) 권장 키

단일 총점만 두지 않고 레이더·설명 카드 재료를 남긴다:

```json
{
  "population_sim": 0.95,
  "matrix_pattern_sim_struct": 0.88,
  "price_level_sim": 0.74,
  "tx_volume_overlap_jaccard": 0.61,
  "weights_applied": { "struct": 0.72, "price": 0.28 },
  "top_matching_cells": [
    { "feature": "2종일반주거×대", "both_share_pct": [0.12, 0.11] },
    { "feature": "자연녹지×답", "both_share_pct": [0.05, 0.06] }
  ]
}
```

실제 저장된 키는 구현 검증 후 확정 가능하며, **사람이 같은 스냅샷 재현에 필요한 버전 문자열**(스케일러 버전 등) 포함을 권장한다.

## 9. 설정 파라미터 요약 예시

| 키(예시) | 의미 | MVP 기본 참고값 |
|-----------|------|-----------------|
| `POP_TOLERANCE_REL` | 인구 상대 허용 | `0.2` |
| `MIN_TX_COUNT_BY_LEVEL` *(맵 형태 가능)* | 최소 거래건수 | 시군구 / 읍면동 등 분리 검토 |
| `COMBINED_WEIGHT_STRUCT` | 구조 cosine 가중 | 예: `0.72` *(튜닝)* |
| `COMBINED_WEIGHT_PRICE` | 가격 유사 가중 | 예: `0.28` *(튜닝)* |
| `TOP_K` | 저장 neighbor 수 | 예: `5` |
| 인접 테이블 버전 필드 | 시도 경계변경 등 | 문자열 또는 날짜 |

## 10. 검증 절차(필수)

정량 순위만 믿지 않고 **사람 검증 패널**을 월별로 일정 시간 수행한다.

**우선 패널 예시**(이름 참고만):

- 오창읍 근처권 특정 레벨
- 세종시 특정 행정단위
- 혁신도시 패널 내 구역 몇 선
- 거래 극소 농외곽 패널(제외 많이 발생하는 지 확인)

**확인 사항**:

- 실제 업무적으로 비슷한가
- 무엇이 유사 신호였는지(상위 차원 이름이 납득되는가)
- 이상 패턴 존재 시 threshold·블록 표본윈도·결합식 조정 회고

## 12. 레포 연계 체크리스트(MVP 시작 전)

다음 레포 규격과 합류 지점 확인 후 첫 브랜치에서 반영 여부 확인:

4. `build_stats`/월간 순서표(`MONTHLY_UPDATE_SOP`)에서 **Profile 빌드 → Twin** 순서 보장
5. parquet 아티팩트 저장 경로(naming 포함)와 로테이션 크기 고려, `profile_version` 포함
